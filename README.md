# Load-balancer---Nginx

# Документация по настройке отказоустойчивого балансировщика нагрузки на базе Nginx

## Введение
Данное руководство описывает процесс создания кластера из двух бэкенд-серверов и балансировщика Nginx с функциями:
- Балансировка нагрузки между серверами.
- Привязка сессий по IP (ip_hash).
- Автоматическое исключение недоступных серверов (health checks).
- Кастомная страница ошибки при отказе всех бэкендов.
- Ограничение частоты запросов (rate limiting) для защиты от перегрузок.

Для реализации потребуется три виртуальные машины с Linux (например, Ubuntu). Пример IP-адресов:
- Backend 1: `192.168.1.10`
- Backend 2: `192.168.1.11`
- Балансировщик: `192.168.1.12` (замените на свои актуальные адреса).

---

## Шаг 1. Подготовка бэкенд-серверов
На каждой из машин, которые будут обрабатывать запросы, установите Nginx и создайте уникальную тестовую страницу для идентификации сервера.

### 1.1 Установка Nginx
```bash
sudo apt update && sudo apt install nginx -y
```

### 1.2 Создание уникальной страницы
- На **Backend 1** (192.168.1.10):
  ```bash
  echo "<h1>Hello from Server 1</h1>" | sudo tee /var/www/html/index.html
  ```
- На **Backend 2** (192.168.1.11):
  ```bash
  echo "<h1>Hello from Server 2</h1>" | sudo tee /var/www/html/index.html
  ```

Убедитесь, что Nginx запущен:
```bash
sudo systemctl start nginx
sudo systemctl enable nginx
```

---

## Шаг 2. Настройка балансировщика
На третьей машине установите Nginx и настройте его в качестве прокси-балансировщика.

### 2.1 Установка Nginx
```bash
sudo apt update && sudo apt install nginx -y
```

### 2.2 Редактирование конфигурации
Очистите стандартный файл конфигурации сайта:
```bash
sudo truncate -s 0 /etc/nginx/sites-available/default
```

Откройте его для редактирования:
```bash
sudo nano /etc/nginx/sites-available/default
```

Вставьте следующую конфигурацию:
```nginx
upstream backend_cluster {
    # Привязка сессии к IP клиента (один и тот же клиент всегда попадает на один бэкенд)
    ip_hash;

    # Бэкенд-серверы с параметрами проверки доступности
    server 192.168.1.10:80 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:80 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;

    # Страница ошибки (если все бэкенды недоступны)
    error_page 502 503 504 /error.html;
    location = /error.html {
        root /usr/share/nginx/html;
        internal;
    }

    location / {
        proxy_pass http://backend_cluster;
        proxy_intercept_errors on;

        # Передача реального IP клиента бэкендам
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header Host $host;
    }
}
```

**Пояснения:**
- `ip_hash` обеспечивает липкость сессии по IP-адресу клиента.
- `max_fails=3` и `fail_timeout=30s` — если сервер не ответит 3 раза подряд, он считается недоступным на 30 секунд.
- `proxy_intercept_errors` позволяет перехватывать ошибки от бэкендов и показывать локальную страницу.

---

## Шаг 3. Создание кастомной страницы ошибки
На балансировщике создайте файл `error.html`, который будет отображаться, когда оба бэкенда недоступны.
```bash
echo "<h1>Извините, сайт временно недоступен. Мы чиним!</h1>" | sudo tee /usr/share/nginx/html/error.html
```

---

## Шаг 4. Проверка и запуск
Проверьте корректность конфигурации:
```bash
sudo nginx -t
```
Если ошибок нет, перезапустите Nginx:
```bash
sudo systemctl restart nginx
```

---

## Шаг 5. Тестирование работы балансировщика

### 5.1 Проверка балансировки
Откройте браузер и перейдите по IP-адресу балансировщика (например, `http://192.168.1.12`). Вы должны увидеть либо «Hello from Server 1», либо «Hello from Server 2». Обновление страницы (без `ip_hash`) показывало бы чередование, но благодаря `ip_hash` с одного и того же IP вы всегда будете попадать на один и тот же сервер.

### 5.2 Имитация отказа бэкенда
Зайдите на **Backend 1** и остановите Nginx:
```bash
sudo systemctl stop nginx
```
Теперь обновите страницу балансировщика. Nginx автоматически перенаправит запросы на **Backend 2** (должна отобразиться страница второго сервера). Если вы остановите и второй сервер, появится кастомная страница ошибки.

## Дополнительная настройка: Rate Limiting (ограничение частоты запросов)
Для защиты бэкенд-серверов от слишком частых запросов (боты, DDoS, сканирование) можно включить ограничение скорости на стороне балансировщика. Nginx использует модуль `ngx_http_limit_req_module`.

### 1. Определение зоны памяти (в блоке `http`)
Отредактируйте основной конфигурационный файл `/etc/nginx/nginx.conf` и внутри блока `http { ... }` добавьте:
```nginx
http {
    # Зона для хранения счетчиков запросов по IP
    limit_req_zone $binary_remote_addr zone=mylimit:10m rate=5r/s;

    # ... остальные настройки
}
```
- `$binary_remote_addr` — ключ (IP клиента) в бинарном виде.
- `zone=mylimit:10m` — имя зоны и размер памяти (10 МБ ≈ 160 000 IP-адресов).
- `rate=5r/s` — разрешено не более 5 запросов в секунду с одного IP.

### 2. Применение ограничения в блоке `server`
В файле конфигурации сайта (например, `/etc/nginx/sites-available/default`) добавьте директивы `limit_req` в нужную локацию:
```nginx
server {
    listen 80;

    # Задаем статус для превышения лимита (по умолчанию 503)
    limit_req_status 429;

    # Страница для ошибки 429
    error_page 429 /too_many_requests.html;

    location / {
        # Применяем ограничение: zone, burst=10, nodelay
        limit_req zone=mylimit burst=10 nodelay;

        proxy_pass http://backend_cluster;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header Host $host;
    }

    location = /too_many_requests.html {
        root /usr/share/nginx/html;
        internal;
    }
}
```
**Параметры:**
- `burst=10` — разрешает кратковременное превышение лимита на 10 запросов (они помещаются в очередь).
- `nodelay` — обрабатывает разрешённые запросы мгновенно, не растягивая их во времени.
- `limit_req_status 429` — изменяет код ошибки на 429 (Too Many Requests), что соответствует стандарту для rate limiting.

### 3. Создание страницы для ошибки 429 (опционально)
```bash
echo "<h1>Слишком много запросов. Пожалуйста, повторите позже.</h1>" | sudo tee /usr/share/nginx/html/too_many_requests.html
```

После изменения конфигурации проверьте её и перезапустите Nginx:
```bash
sudo nginx -t && sudo systemctl restart nginx
```

### Как это работает
- При нормальной нагрузке пользователь не замечает ограничений.
- Если клиент отправляет более 5 запросов в секунду, первые 5 обрабатываются, следующие 10 попадают в очередь (`burst`), а все остальные сразу получают ошибку 429.
- Благодаря этому бэкенд-серверы освобождаются от избыточного трафика.

Для тестирования ограничений (Rate Limiting) лучше всего использовать утилиту Apache Benchmark (ab). Она умеет имитировать наплыв пользователей, отправляя сотни запросов в секунду.
1. Установите инструмент
Если его еще нет на вашем балансировщике (или на любой другой машине, с которой будете бить по IP балансировщика):

sudo apt update && sudo apt install apache2-utils -y

2. Запустите атаку (тест)
Выполните команду, подставив IP вашего балансировщика:

# -n 100 — всего 100 запросов# -c 5   — в 5 потоков одновременно
ab -n 100 -c 5 http://IP_ВАШЕГО_БАЛАНСИРОВЩИКА/

3. На что смотреть в результате?
В консоли вы увидите подробный отчет. Ищите раздел "Complete requests":

* Non-2xx responses: Здесь будет число запросов, которые получили ошибку 429 (или 503, если не меняли статус).
* Если ваш лимит rate=5r/s и burst=10, то из 100 быстрых запросов примерно 85 штук должны провалиться (Nginx их отсечет).

4. Проверьте логи в реальном времени
Пока идет тест, на балансировщике выполните:

tail -f /var/log/nginx/error.log

Вы увидите строки: limiting requests, excess: ... by zone "mylimit". Это подтверждение того, что защита сработала.
------------------------------
Что мы имеем в итоге:

   1. Балансировщик распределяет нагрузку между ВМ.
   2. IP Hash держит сессии пользователей на одной ВМ.
   3. Health Checks выключают упавшие ВМ из пула.
   4. Rate Limiting защищает от перегрузки и ботов.
   5. Логи показывают, куда именно улетел запрос.

## Заключение
Вы настроили полноценный балансировщик Nginx с отказоустойчивостью, липкостью сессий и защитой от перегрузок. Такая конфигурация подходит для большинства веб-приложений, обеспечивая высокую доступность и безопасность. При необходимости вы можете расширить кластер, добавив новые бэкенд-серверы в блок `upstream`.
