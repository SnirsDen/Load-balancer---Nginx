# Настройка отказоустойчивого балансировщика нагрузки на базе Nginx

Данное руководство описывает пошаговую настройку Nginx в качестве балансировщика нагрузки для двух и более бэкенд-серверов. Вы также научитесь ограничивать частоту запросов (rate limiting), настраивать резервные серверы и обрабатывать ошибки.

---

## 1. Подготовка бэкенд-серверов (ВМ1, ВМ2, …)

На каждой рабочей машине, которая будет обрабатывать запросы, должен быть запущен веб-сервер. В примере используется Nginx на порту `8080`.

### 1.1 Установка Nginx
```bash
sudo apt update
sudo apt install nginx -y
```

### 1.2 Идентификация серверов
Чтобы различать, какой сервер ответил, создайте на каждом уникальный файл `index.html`:

* **ВМ 1** (192.168.114.146)
  ```bash
  echo "Server 1" | sudo tee /var/www/html/index.html
  ```
* **ВМ 2** (192.168.114.147)
  ```bash
  echo "Server 2" | sudo tee /var/www/html/index.html
  ```
* **ВМ 3 (резервный)** (192.168.114.148)
  ```bash
  echo "Server 3 (backup)" | sudo tee /var/www/html/index.html
  ```

### 1.3 Настройка порта
Убедитесь, что веб-серверы слушают порт `8080` (или тот, который будет указан в конфигурации балансировщика). Для Nginx измените файл `/etc/nginx/sites-available/default`:
```nginx
server {
    listen 8080 default_server;
    listen [::]:8080 default_server;
    root /var/www/html;
    ...
}
```
Затем перезапустите Nginx:
```bash
sudo systemctl restart nginx
```

## 2. Глобальная настройка балансировщика (`/etc/nginx/nginx.conf`)

В этом файле задаются общие параметры: лимиты запросов, формат логов, подключение дополнительных конфигураций.

```nginx
http {
    # 1. Лимиты запросов (Rate Limiting)
    # 10m — размер памяти для хранения состояний IP-адресов
    # 30r/s — максимальная частота запросов с одного IP
    limit_req_zone $binary_remote_addr zone=mylimit:10m rate=30r/s;

    # 2. Расширенный формат логов для видимости, какой бэкенд ответил
    log_format upstream_log '$remote_addr - $remote_user [$time_local] '
                           '"$request" $status $body_bytes_sent '
                           'to: $upstream_addr | time: $upstream_response_time';

    # 3. Применяем этот формат ко всем запросам
    access_log /var/log/nginx/access.log upstream_log;

    # 4. Стандартные подключения
    include /etc/nginx/mime.types;
    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}
```

---

## 3. Настройка логики балансировки (`/etc/nginx/sites-available/default`)

Здесь описывается группа бэкенд-серверов (апстрим) и правила обработки входящих запросов.

```nginx
# Группа серверов (кластер)
upstream backend_cluster {
    # Раскомментируйте строку ниже, если требуется привязка сессий к одному серверу
    # ip_hash;

    # Основные серверы
    server 192.168.114.146:8080 max_fails=3 fail_timeout=30s;
    server 192.168.114.147:8080 max_fails=3 fail_timeout=30s;

    # Резервный сервер (включится, если основные недоступны)
    server 192.168.114.148:8080 backup;
}

server {
    listen 80;

    # Пользовательские страницы ошибок
    error_page 502 503 504 /error.html;
    error_page 429 /too_many_requests.html;

    location / {
        # Применение ограничения частоты запросов
        # burst=50 — размер очереди, nodelay — отвечать сразу без задержки
        limit_req zone=mylimit burst=50 nodelay;

        # Перенаправление запроса в группу бэкендов
        proxy_pass http://backend_cluster;

        # Передача реальных данных клиента бэкендам
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # Включаем перехват ошибок от бэкендов для показа своих страниц
        proxy_intercept_errors on;
    }

    # Внутренние location для страниц ошибок
    location = /error.html {
        root /usr/share/nginx/html;
        internal;
    }

    location = /too_many_requests.html {
        root /usr/share/nginx/html;
        internal;
    }
}
```

### Пояснения
* `max_fails=3 fail_timeout=30s` — если сервер не ответит 3 раза за 30 секунд, он считается недоступным.
* `backup` — сервер используется только когда основные недоступны.
* `proxy_intercept_errors` — позволяет Nginx обрабатывать ошибки бэкендов (например, 502) и показывать свои страницы.

## 4. Создание страниц-заглушек

На самом балансировщике создайте HTML-файлы, которые будут показываться при ошибках или превышении лимита запросов.

```bash
# Страница для ошибок 502, 503, 504
echo "<h1>Серверы перегружены или недоступны</h1>" | sudo tee /usr/share/nginx/html/error.html

# Страница для ошибки 429 (Too Many Requests)
echo "<h1>Слишком много запросов. Подождите.</h1>" | sudo tee /usr/share/nginx/html/too_many_requests.html
```

## 5. Применение и проверка конфигурации

После любых изменений выполняйте следующие команды:

1. **Проверка синтаксиса**
   ```bash
   sudo nginx -t
   ```
   Ожидаемый вывод: `syntax is ok` и `test is successful`.

2. **Перезапуск Nginx**
   ```bash
   sudo systemctl restart nginx
   ```

3. **Мониторинг логов в реальном времени**
   ```bash
   tail -f /var/log/nginx/access.log
   ```
   Пример строки лога:
   ```
   192.168.1.100 - - [21/May/2025:12:00:00 +0000] "GET / HTTP/1.1" 200 12 to: 192.168.114.146:8080 | time: 0.005
   ```
   Здесь видно, какой бэкенд обработал запрос (`to: 192.168.114.146:8080`) и время ответа.

## 6. Устранение возможных проблем (траблшутинг)

| Проблема            | Возможная причина                                | Решение                                                                 |
|---------------------|--------------------------------------------------|-------------------------------------------------------------------------|
| **Ошибка 502**      | Бэкенд-сервер не отвечает или недоступен         | Проверьте firewall на бэкенде (`sudo ufw allow 8080`). Убедитесь, что Nginx запущен и слушает нужный порт. |
| **Ошибка 429**      | Сработало ограничение частоты запросов           | Либо увеличьте `rate` в `nginx.conf`, либо проверьте, не слишком ли часто обновляется страница. |
| **Медленная работа**| Бэкенд долго обрабатывает запросы                | Посмотрите в логах параметр `time`: если он большой (>1 сек), проблема в самом бэкенде. Проверьте его нагрузку. |
| **Неравномерное распределение** | Используется `ip_hash` или другой метод балансировки | Убедитесь, что метод распределения (по умолчанию round-robin) соответствует вашим требованиям. |

## Заключение и дополнительная возможность (SSL)

Вы настроили отказоустойчивый балансировщик с базовыми функциями. Теперь трафик распределяется между двумя основными серверами, есть резервный сервер, настроены лимиты и пользовательские страницы ошибок.

**Нужно ли добавить раздел по автоматическому выпуску SSL-сертификата (HTTPS) через Certbot?**

SSL-шифрование — важный элемент безопасности современных веб-приложений. Добавление HTTPS защищает данные пользователей и повышает доверие к ресурсу. Certbot позволяет легко получить и автоматически обновлять бесплатные сертификаты Let's Encrypt.

Если вы планируете использовать балансировщик для публичного сайта, настоятельно рекомендуется настроить HTTPS. В этом случае в конфигурацию Nginx потребуется добавить:
* Прослушивание порта 443 (SSL).
* Пути к сертификатам (полученным через Certbot).
* Возможно, перенаправление с HTTP на HTTPS.
